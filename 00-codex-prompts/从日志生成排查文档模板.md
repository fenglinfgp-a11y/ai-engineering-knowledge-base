# 从日志生成排查文档模板

来源类型：实战总结

## 问题场景

已经有脱敏日志，需要让 Codex 归纳成可复用的排查文档。

## 适用范围

适用于 Nginx、systemd、Spring Boot、PHP、MySQL、Android logcat、Gradle 构建等日志分析。

## 判断方法

- 日志中没有真实 IP、域名、账号、Token、客户资料。
- 日志只用于定位合法开发、部署、兼容性、配置或运行时问题。
- 排查动作以只读命令和最小变更为主。

## 操作步骤

可复制给 Codex：

```text
请只在 /Users/dora/Documents/ai-engineering-knowledge-base 中操作。

以下是已脱敏日志，请总结成排查文档：
<粘贴脱敏日志>

请完成：
1. 判断日志中的核心错误。
2. 给出可能原因和排查顺序。
3. 生成一篇文档或案例，使用占位符，不写真实敏感信息。
4. 每条命令都说明用途，危险命令标注风险。
5. 必须写清楚“怎么验证修好了”。
6. 更新相关索引。
7. 执行 find . -name "*.md" -size 0 和 git status。

禁止：
- 不要输出绕过登录、支付、授权、风控的方案。
- 不要保存真实密钥、Token、客户信息、数据库真实数据。
```

## 验证方法

- 文档中有“关键日志”和“验证方式”。
- 命令解释清楚用途。
- 敏感字段全部替换为 `<PLACEHOLDER>`。

## 注意事项

- 如果日志不足以判断原因，应标注“待补充”，不要编造结论。
- 生产环境操作前必须先备份或使用只读命令确认影响范围。

## 相关命令

```bash
grep -n "<keyword>" <log-file>
```

用途：在脱敏日志中定位关键错误。`<log-file>` 必须是本地脱敏文件。

```bash
git diff --check
```

用途：检查 Markdown 修改中是否有多余空白问题。
