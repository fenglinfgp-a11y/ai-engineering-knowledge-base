# 最小改动模板

来源类型：实战总结

## 问题场景

项目已有稳定逻辑，只想修一个 bug 或补一个小功能，避免 Codex 顺手重构。

## 判断方法

- 是否只涉及一个明确问题。
- 是否可以用现有结构解决。
- 是否不需要引入新框架或大改目录。

## 操作步骤

可复制模板：

```text
请用最小改动完成这个任务。

要求：
1. 优先沿用现有代码风格和工具。
2. 不做无关重构。
3. 不升级依赖，除非这是修复必需项。
4. 不改公共接口，除非先说明影响。
5. 修改后运行最相关的测试或构建命令。

任务：
<写具体问题>
```

## 验证方法

- 变更集中在少量相关文件。
- 没有无关格式化大面积 diff。
- 验证命令能覆盖本次修改。

## 注意事项

- 如果问题根因需要大改，先要求 Codex 给方案，不要让它直接重构。

## 适用范围

Codex 指令模板、任务边界控制和 AI 辅助维护流程。

## 相关命令

`pwd`、`git status --short`、`git diff --stat`。
