# 让 Codex 做部署排查模板

来源类型：实战总结 / 我的项目经验

## 问题场景

服务部署后不可访问、接口 500、Nginx 502、端口不通，希望 Codex 先做只读排查，再提出最小变更建议。

## 适用范围

- 适用于知识库内的部署排查指导和脱敏日志分析。
- 适用于 Ubuntu、Nginx、Docker、Spring Boot、PHP、MySQL 通用问题。
- 不适用于让 Codex 登录服务器、输入密码或操作业务项目。

## 判断方法

- 先确认问题层级：DNS、端口、防火墙、Nginx、后端、数据库。
- 先只读命令，再高风险操作。
- 高风险操作必须说明影响和回滚方式。

## 操作步骤

可直接复制以下提示词：

```text
请只基于我提供的脱敏信息和知识库进行部署排查建议。
不要保存服务器密码、Token、密钥、客户资料、真实数据库数据。
不要自动执行高风险命令。

问题现象：【现象】
服务器系统：【例如 Ubuntu 版本，可脱敏】
服务类型：【Nginx / Docker / Spring Boot / PHP-FPM / MySQL】
脱敏日志：【贴脱敏日志】

请输出：
1. 最可能的问题层级。
2. 建议先执行的只读命令，每条说明用途。
3. 需要谨慎执行的危险命令和风险。
4. 如何验证问题是否恢复。
5. 可沉淀到知识库的文档路径建议。
```

用途：让 Codex 按排查顺序分析部署问题。

## 验证方法

- 推荐命令优先只读。
- 高风险命令有风险说明。
- 输出没有要求记录真实密码、Token 或客户数据。

## 注意事项

- 生产环境重启、删除、覆盖、恢复数据库都必须人工确认。
- 贴日志前先脱敏。
- 如果要写入知识库，应另用“沉淀案例模板”。

## 相关命令

`ss -lntp`、`curl -i`、`nginx -t`、`systemctl status`、`journalctl -u`、`docker logs`。
