# 让 Codex 沉淀案例模板

来源类型：实战总结 / 我的项目经验

## 问题场景

一个问题已经排查完成，希望把过程脱敏后沉淀成 `cases/` 下的可复用案例。

## 适用范围

- 适用于脱敏后的日志分析、部署问题、数据库问题、交付问题。
- 适用于总结现象、判断方法、命令顺序和验证方式。
- 不适用于保存客户真实日志、真实数据和密钥。

## 判断方法

- 案例必须能复用，不能依赖客户专有信息。
- 先替换域名、IP、账号、库名、Token、订单号。
- 只保留通用错误现象和排查流程。

## 操作步骤

可直接复制以下提示词：

```text
请只在 /Users/dora/Documents/ai-engineering-knowledge-base 内操作。
不要读取或修改业务项目。
我提供的是已脱敏排查过程，请沉淀为 cases/ 下的案例。

案例主题：【主题】
现象：【脱敏现象】
关键日志：【脱敏日志】
最终原因：【原因】
修复方式：【修复方式】

要求：
1. 不保存真实域名、IP、账号、Token、客户资料、真实数据库数据。
2. 文档包含问题场景、判断方法、操作步骤、验证方法、注意事项、相关命令。
3. 命令必须解释用途，危险命令标记风险。
4. 更新 99-index/技术索引.md 和 99-index/问题排查索引.md。
5. 执行 find . -name "*.md" -size 0 和 git status。
```

用途：把一次问题处理过程变成脱敏案例。

## 验证方法

- 案例没有真实敏感信息。
- 案例能从索引中找到。
- 案例保留了复现、判断、操作和验证链路。

## 注意事项

- 原始日志如果没脱敏，不要直接贴。
- 不确定能否公开的细节写“待确认”，不要写具体配置。

## 相关命令

`rg`、`find . -name "*.md" -size 0`、`git diff --stat`、`git status`。
