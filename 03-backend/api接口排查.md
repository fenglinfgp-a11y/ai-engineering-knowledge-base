# API 接口排查

来源类型：实战总结

## 问题场景

接口返回 404、401、403、500、超时，或前端说“接口不通”。

## 判断方法

- 先确认 URL、方法、请求头、参数是否正确。
- 再确认服务端路由是否存在。
- 再看后端日志和数据库日志。
- 最后排查 Nginx、网关、跨域、证书。

## 操作步骤

```bash
curl -i http://127.0.0.1:8080/health
```

用途：查看接口响应头、状态码和响应体。

```bash
curl -X POST http://127.0.0.1:8080/api/demo -H "Content-Type: application/json" -d '{"id":1}'
```

用途：用 JSON 请求复现 POST 接口问题。

```bash
tail -n 200 logs/app.log
```

用途：查看应用最近 200 行日志。风险：日志可能包含敏感信息，外发前必须脱敏。

## 验证方法

- 同一请求返回预期状态码。
- 后端日志不再出现同类异常。
- 前端 Network 面板中的请求和直接 `curl` 结果一致。

## 注意事项

- 401 通常是认证问题，403 通常是权限问题，500 通常是服务端异常。
- 不要把真实用户 Token 写入知识库或聊天记录。

## 适用范围

后端服务结构、接口排查、日志分析和部署验证。

## 相关命令

`curl -i`、`tail -n`、`journalctl`、`systemctl status`。
