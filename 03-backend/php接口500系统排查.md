# PHP 接口 500 系统排查

来源类型：官方文档 / 实战总结  
参考来源：PHP 官方文档、Nginx 官方文档

## 问题场景

PHP 接口返回 500，页面白屏，Nginx 访问日志只有 500，业务方只知道“接口挂了”。

## 适用范围

- 适用于 PHP-FPM + Nginx 常见部署。
- 适用于 Laravel、ThinkPHP、原生 PHP 等通用日志排查。
- 不记录客户请求参数、真实数据库内容或生产密钥。

## 判断方法

- 用 `curl -i` 确认状态码和响应头。
- 查 Nginx error log，判断是否请求已到 PHP-FPM。
- 查 PHP-FPM 日志或应用日志，找到第一处异常。
- 检查 PHP 扩展、文件权限、配置缓存、数据库连接。
- 区分语法错误、运行时异常、超时和上游不可用。

## 操作步骤

```bash
curl -i http://127.0.0.1/<api-path>
```

用途：从服务器本机复现接口响应，排除外网链路影响。

```bash
tail -n 200 /var/log/nginx/error.log
```

用途：查看 Nginx 最近错误日志。

```bash
tail -n 200 /var/log/php*-fpm.log
```

用途：查看 PHP-FPM 最近错误日志，路径按系统实际版本调整。

```bash
php -m
```

用途：列出已安装 PHP 扩展，排查缺少 `pdo_mysql`、`mbstring` 等依赖。

```bash
php -l /path/to/test.php
```

用途：检查单个 PHP 文件语法。风险：只能检查语法，不能证明业务逻辑正确。

## 验证方法

- 同一个脱敏请求能稳定返回预期状态码。
- Nginx error log 不再出现 FastCGI 或权限错误。
- PHP-FPM 日志中不再出现同类 fatal error。
- 应用日志能记录可解释的业务错误，而不是白屏 500。

## 注意事项

- 生产环境不要打开面向公网的详细错误显示。
- `.env`、数据库密码、客户请求体不能写入知识库。
- 修改权限前先确认目录归属，避免为了修复 500 过度放权。

## 相关命令

`curl -i`、`tail -n`、`php -m`、`php -l`、`systemctl status php*-fpm`。
