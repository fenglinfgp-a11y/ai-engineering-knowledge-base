# 后端日志排查

来源类型：实战总结

## 问题场景

服务异常但页面只显示“系统错误”，需要从日志定位真实原因。

## 判断方法

- 先按时间定位错误。
- 再按请求 ID、用户 ID、订单号等脱敏线索关联日志。
- 优先看第一处异常栈，不只看最后一行。

## 操作步骤

```bash
tail -f logs/app.log
```

用途：实时查看应用日志。风险：生产日志可能包含敏感信息，不要直接复制外发。

```bash
grep -n "ERROR" logs/app.log | tail -50
```

用途：查看最近 50 条错误行及行号。

```bash
journalctl -u <service-name> -n 200 --no-pager
```

用途：查看 systemd 服务最近 200 行日志，不进入分页器。

## 验证方法

- 修复后重复请求，日志不再出现同类错误。
- 请求 ID 在访问日志和应用日志中能对应。
- 错误率下降或监控恢复正常。

## 注意事项

- `journalctl --vacuum-time=7d` 会清理旧日志，风险是删除历史排查证据，执行前确认保留周期。
- 知识库只能保存脱敏示例日志。

## 适用范围

后端服务结构、接口排查、日志分析和部署验证。

## 相关命令

`curl -i`、`tail -n`、`journalctl`、`systemctl status`。
