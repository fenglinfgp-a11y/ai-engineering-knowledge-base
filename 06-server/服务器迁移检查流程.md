# 服务器迁移检查流程

来源类型：官方文档 / 实战总结 / 我的项目经验  
参考来源：Ubuntu Server、Nginx、Docker、MySQL 官方文档

## 问题场景

需要把服务从旧服务器迁移到新服务器，或交付前确认新环境是否具备运行条件。

## 适用范围

- 适用于 Web 服务、Nginx、Java/PHP 后端、MySQL、Docker 的通用迁移检查。
- 适用于脱敏配置清单和测试环境演练。
- 不保存真实服务器密码、私钥、Token、客户数据库数据。

## 判断方法

- 先列清单：域名、端口、服务、数据库、文件、定时任务、证书。
- 新服务器先做只读环境检查，再导入测试数据。
- 迁移前确认备份和回滚路径。
- 切换域名或流量前先用 hosts 或内网地址验证。
- 切换后观察日志、状态码和核心业务流程。

## 操作步骤

```bash
uname -a
```

用途：查看服务器系统和内核信息。

```bash
df -h
```

用途：查看磁盘空间，确认迁移目标容量足够。

```bash
ss -lntp
```

用途：查看新服务器监听端口。

```bash
systemctl list-units --type=service --state=running
```

用途：查看正在运行的系统服务。

```bash
sudo nginx -t
```

用途：检查 Nginx 配置。

```bash
mysqldump -h <host> -u <user> -p --no-data <database_name> > schema.sql
```

用途：导出结构 SQL 供迁移检查。风险：生产库操作前确认权限和目标，备份文件不得提交到 Git。

## 验证方法

- 新服务器端口、服务、证书、数据库连接均正常。
- 核心接口和页面在新环境通过。
- Nginx、应用、数据库日志没有新增异常。
- DNS 或流量切换后仍可回滚到旧环境。

## 注意事项

- 迁移不是只复制文件，还包括环境变量、定时任务、证书、服务配置和数据库结构。
- 真实 `.env`、私钥、生产备份不能写入知识库。
- 切换 DNS 前记录 TTL 和回滚方案。

## 相关命令

`uname -a`、`df -h`、`ss -lntp`、`systemctl list-units`、`nginx -t`、`mysqldump --no-data`。
