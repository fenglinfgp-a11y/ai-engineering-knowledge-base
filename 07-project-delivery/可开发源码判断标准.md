# 可开发源码判断标准

来源类型：实战总结 / 我的项目经验

## 问题场景

收到一份源码后，需要判断它是可继续开发的源码，还是只有构建产物或不完整文件。

## 判断方法

- 是否有依赖声明：`package.json`、`pom.xml`、`composer.json` 等。
- 是否有源码目录：`src/`、`app/`、`routes/`、`controllers/`。
- 是否有构建或启动脚本。
- 是否有数据库结构。

## 操作步骤

```bash
find . -maxdepth 3 -name "package.json" -o -name "pom.xml" -o -name "composer.json"
```

用途：查找常见依赖声明文件。

```bash
find . -maxdepth 3 -type d \\( -name src -o -name app -o -name controllers -o -name routes \\)
```

用途：查找常见源码目录。

```bash
find . -type f \\( -name "*.sql" -o -name "*migration*" \\) | head
```

用途：查找数据库结构或迁移文件。

## 验证方法

- 能根据依赖文件安装依赖。
- 能找到业务源码入口。
- 能在测试环境启动或构建。
- 数据库结构能初始化。

## 注意事项

- 只有 `dist/`、`build/`、`target/` 往往不是完整可开发源码。
- 不要把缺失依赖的二进制文件当成源码交付。

## 适用范围

源码交付、交付包清理、可开发性判断和客户验收准备。

## 相关命令

`find`、`du -sh`、`git status --short`、敏感字段扫描命令。
