# 如何判断后端是不是源码

来源类型：官方文档 / 实战总结 / 我的项目经验  
参考来源：Spring Boot、PHP、Docker 官方文档

## 问题场景

收到后端交付包后，需要判断它是可继续开发的源码，还是只有 jar、war、dist、镜像或运行产物。

## 适用范围

- 适用于 Java/Spring Boot、PHP、Node 等后端交付包初步验收。
- 适用于公开工程结构知识和脱敏交付检查。
- 不读取或保存业务项目源码细节、客户资料和密钥。

## 判断方法

- Java 源码通常包含 `pom.xml`、`build.gradle`、`src/main/java`。
- PHP 源码通常包含 `composer.json`、`app/`、`public/`、框架配置示例。
- Node 后端通常包含 `package.json`、`src/`、启动脚本。
- 只有 `.jar`、`.war`、`target/`、`build/`、镜像 tar 时通常不是完整源码。
- 可开发源码应能安装依赖、运行测试或启动本地服务。

## 操作步骤

```bash
find . -maxdepth 3 -type f | sort
```

用途：查看交付包顶层和二级文件结构。

```bash
find . -maxdepth 3 -type f \\( -name "pom.xml" -o -name "build.gradle" -o -name "composer.json" -o -name "package.json" \\)
```

用途：查找后端依赖描述文件。

```bash
find . -maxdepth 4 -type d \\( -name "src" -o -name "app" -o -name "config" \\)
```

用途：查找常见源码目录。

```bash
jar tf <app>.jar | head
```

用途：查看 jar 内部结构。风险：只能判断运行包结构，不能代表交付源码完整。

## 验证方法

- 能找到源码目录和依赖文件。
- README 或交付说明写清安装依赖、配置示例、启动命令。
- 在测试环境能完成构建或启动。
- 没有把真实 `.env`、生产日志、客户数据作为源码交付。

## 注意事项

- 有源码不等于可运行，还要看依赖、配置、数据库结构和启动说明。
- 运行产物可以作为部署附件，但不能替代源码交付。
- 只记录通用判断方法，不复制客户项目代码。

## 相关命令

`find`、`jar tf`、`du -sh`、`git status --short`。
