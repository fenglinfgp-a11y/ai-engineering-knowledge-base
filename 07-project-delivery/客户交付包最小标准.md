# 客户交付包最小标准

来源类型：实战总结 / 我的项目经验  
参考来源：Spring Boot、Vue、Nuxt、MySQL、Docker 官方文档

## 问题场景

项目准备交付，需要确认交付包是否足够接手方部署、运行、二次开发和排查问题。

## 适用范围

- 适用于前端、后端、数据库三件套交付。
- 适用于源码交付、自测交付和部署交接。
- 不包含生产密钥、客户资料、真实数据库数据。

## 判断方法

- 后端：源码、依赖文件、配置示例、启动说明、部署说明。
- 前端：源码、依赖文件、构建说明、部署目录说明。
- 数据库：结构 SQL、迁移说明、脱敏样例数据说明。
- 运维：服务端口、Nginx、systemd、Docker、日志路径说明。
- 安全：敏感信息已清理，真实数据不在包内。

## 操作步骤

```bash
find . -maxdepth 3 -type f | sort
```

用途：查看交付包结构是否完整。

```bash
du -sh *
```

用途：查看目录大小，发现异常大的日志、缓存、依赖目录。

```bash
find . -type f \\( -name ".env" -o -name "*.key" -o -name "*.pem" -o -name "*.sql" -o -name "*.log" \\)
```

用途：扫描常见敏感或高风险文件，后续逐个判断是否允许交付。

```bash
git status --short
```

用途：确认交付前工作区状态，避免临时文件混入。

## 验证方法

- 新环境能按 README 安装依赖并启动后端。
- 前端能构建并访问测试接口。
- 数据库结构能导入空测试库。
- 敏感文件扫描结果已逐项处理。

## 注意事项

- 最小标准不是“越多越好”，生产备份、日志、密钥不能交付。
- 交付包应包含 `.env.example`，不能包含真实 `.env`。
- 如果只交付运行包，需要明确不具备二次开发能力。

## 相关命令

`find`、`du -sh`、`git status --short`、`mysqldump --no-data`。
