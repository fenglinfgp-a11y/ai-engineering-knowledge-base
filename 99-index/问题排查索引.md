# 问题排查索引

来源类型：实战总结

## 问题场景

遇到线上或测试环境问题时，需要按层级排查，避免一上来就改配置。

## 判断方法

通用顺序：

1. 复现问题。
2. 确认环境。
3. 看日志。
4. 看配置。
5. 做最小变更。
6. 验证是否恢复。
7. 脱敏后归档到 `cases/`。
8. 更新索引并提交。

## 操作步骤

### 服务无法访问

1. `ss -lntp` 查看端口是否监听。
2. `systemctl status <service-name>` 查看服务是否运行。
3. `nginx -t` 检查代理配置。
4. `curl -i http://127.0.0.1:<port>` 验证本机服务。
5. 需要案例模板时看 `cases/nginx-502排查案例.md`。
6. HTTPS 证书问题看 `cases/server/nginx-https证书排查案例.md`。
7. 端口不通看 `cases/server/ubuntu端口不通排查案例.md`。
8. 容器启动失败看 `cases/server/docker容器启动失败案例.md`。
9. 需要记录到 GitHub Issue 时看 `.github/ISSUE_TEMPLATE/服务器部署记录.md`。

### 接口 500

1. 用 `curl -i` 复现请求。
2. 查 `logs/app.log` 或 `journalctl`。
3. 查看数据库连接和 SQL 异常。
4. 修复后重复同一请求验证。
5. PHP 接口 500 看 `cases/backend/php接口500排查案例.md`。
6. PHP-FPM 异常看 `cases/backend/php-fpm异常排查案例.md`。
7. 需要形成排查记录时看 `.github/ISSUE_TEMPLATE/bug排查记录.md`。

### 前端白屏

1. 打开浏览器 Console。
2. 查看 Network 是否有 JS/CSS 404。
3. 本地运行 `npm run build`。
4. 用静态服务预览 `dist/`。

### 数据库异常

1. 先备份。
2. 用 `SHOW CREATE TABLE` 看结构。
3. 用 `EXPLAIN` 看慢 SQL。
4. 删除或清理前先 `SELECT COUNT(*)` 验证范围。
5. 日志表清理模板看 `cases/mysql日志表清理案例.md`。
6. 慢 SQL 看 `cases/database/mysql慢SQL-explain案例.md`。
7. 备份恢复看 `cases/database/mysql备份恢复案例.md`。
8. 日志表过大看 `cases/database/mysql日志表过大清理案例.md`。
9. 字段缺失看 `cases/database/mysql字段缺失排查案例.md`。

### Spring Boot 启动失败

1. 用 `java -version` 确认 JDK。
2. 前台运行 `java -jar` 查看完整启动日志。
3. 用 `ss -lntp` 检查端口。
4. 用 `journalctl` 查看 systemd 日志。
5. 案例模板看 `cases/springboot启动失败案例.md`。
6. systemd 部署看 `cases/backend/springboot-systemd部署案例.md`。
7. 端口占用看 `cases/backend/springboot端口占用案例.md`。

### Android 闪退

1. 用 `adb devices` 确认测试设备。
2. 用 `adb logcat -c` 清空旧日志。
3. 复现后保存 `adb logcat -v time`。
4. 查找 `FATAL EXCEPTION` 和第一处业务代码堆栈。
5. 案例模板看 `cases/android日志分析案例.md`。
6. ANR 看 `cases/android/android-anr日志分析案例.md`。
7. Gradle 构建失败看 `cases/android/android-gradle构建失败案例.md`。
8. LSPosed 模块未加载看 `cases/android/lsposed模块未加载排查案例.md`。
9. APK 安装失败看 `cases/android/apk安装失败排查案例.md`。
10. Android 调试记录模板看 `.github/ISSUE_TEMPLATE/android调试记录.md`。

### 交付包不完整

1. 用 `find . -maxdepth 3 -type f | sort` 查看结构。
2. 查找 `package.json`、`pom.xml`、`composer.json` 等依赖文件。
3. 查找 `.env`、私钥、SQL、日志等风险文件。
4. 按 `checklists/源码交付前检查清单.md` 验收。
5. 案例模板看 `cases/前后端数据库交付包检查案例.md`。
6. 前端只有 dist 看 `cases/delivery/前端dist不是源码判断案例.md`。
7. 后端只有 jar 看 `cases/delivery/jar不是后端源码判断案例.md`。
8. 交付包清理看 `cases/delivery/客户交付包清理案例.md`。
9. 三件套检查看 `cases/delivery/前后端数据库三件套检查案例.md`。
10. 源码交付记录模板看 `.github/ISSUE_TEMPLATE/源码交付记录.md`。

### Nginx 502

1. 本机访问后端健康接口。
2. 检查 upstream 地址和端口。
3. 查看 Nginx error log。
4. 执行 `nginx -t` 后再 reload。
5. 专项案例看 `cases/server/nginx反向代理502案例.md`。
6. 完整流程看 `06-server/nginx反向代理完整排查.md`。

1. 新问题先用 `.github/ISSUE_TEMPLATE/知识补充任务.md` 或对应排查记录模板记录背景。

3. 写入知识库前先看 `security/脱敏规则.md`。
4. 不确定是否允许写入时看 `security/禁止写入知识库的内容.md`。
5. 命令风险判断看 `security/命令风险分级.md`。

## 资料来源排查

- 新增文档前先看 `sources.md`。
- 不确定是否合法或是否公开时，不写具体操作，只保留待确认说明。

## 验证方法

- 用户路径可正常完成。
- 日志不再出现同类错误。
- 监控、状态码、页面表现与预期一致。

## 注意事项

- 生产环境先只读排查，再做变更。
- 每次只改一个变量，便于判断是否真正修复。
- 所有真实日志、数据库内容、客户信息必须脱敏。
