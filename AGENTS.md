# Codex 维护规则

本仓库是 AI 技术知识库，不是业务项目。Codex 在维护本仓库时必须遵守以下规则。

## 工作边界

- 每次操作前必须执行或确认 `pwd`，当前目录必须是 `/Users/dora/Documents/ai-engineering-knowledge-base`。
- 只允许修改本仓库目录：`ai-engineering-knowledge-base`。
- 不要修改用户其他业务项目，除非用户单独明确要求。
- 默认只分析和维护本知识库，不碰业务项目、不读取业务项目文件。
- 不生成业务源码、不复制闭源项目代码、不搬运完整第三方项目源码。
- 不保存服务器密码、Token、密钥、客户资料、License 私钥、数据库真实数据。
- 不写绕过登录、绕过支付、绕过授权、绕过风控、伪造订单、攻击第三方系统的内容。

## 内容来源

每篇文档必须标注来源类型，允许值：

- 官方文档
- 开源项目
- 开源社区资料
- 实战总结
- 我的项目经验

如果引用具体资料，优先记录官方文档链接或开源仓库链接。禁止大段复制原文，必须总结成可查的工程笔记。

## 文档结构

技术文档优先使用以下结构：

1. 问题场景
2. 判断方法
3. 操作步骤
4. 验证方法
5. 注意事项

命令必须附带用途说明。危险命令必须明确写出风险，例如删除数据、覆盖文件、重启服务、中断连接。

## 安全要求

- 示例中的域名、IP、用户名、数据库名必须使用占位符。
- 示例密钥必须使用 `<TOKEN>`、`<PASSWORD>`、`<SECRET>` 等占位符。
- 数据库示例只能使用测试库、测试表、脱敏字段。
- 任何新增案例必须脱敏，只能保留通用现象、判断方法、命令顺序、验证方式和可复用经验。
- 涉及 Android Billing、iOS 插件、Hook、日志分析时，只允许记录合法开发、授权测试、兼容性调试和排错内容。

## 提交流程建议

1. 新增文档前先检查是否已有同类主题。
2. 新文档放到最接近的目录。
3. 更新相关索引文件。
4. 修改后必须更新索引：`99-index/技术索引.md`、必要时更新 `99-index/常用命令索引.md` 和 `99-index/问题排查索引.md`。
5. 修改后必须执行 Markdown 空文件检查：`find . -name "*.md" -size 0`。
6. 提交前必须输出 `git status`，确认只包含知识库相关改动。
7. 运行 `find . -type f | sort` 检查结构。
8. 不确定来源或合法性时，先标注为待确认，不写具体操作。

## 问题场景

Codex 后续维护本知识库时，可能会误把它当成业务项目，或者写入敏感信息、违法绕过类内容。

## 判断方法

- 当前工作目录必须是本知识库。
- 文档内容必须能归类为公开资料、官方文档、开源经验或脱敏实战总结。
- 涉及真实项目时，只能总结方法，不能复制业务代码和敏感数据。

## 操作步骤

```bash
pwd
```

用途：确认当前工作目录，避免误改其他项目。

```bash
find . -type f | sort
```

用途：查看知识库文件清单，确认新增文档位置。

```bash
git diff --stat
```

用途：查看本次修改影响范围，避免无关改动混入。

```bash
find . -name "*.md" -size 0
```

用途：检查是否存在空 Markdown 文档。

```bash
git status
```

用途：提交前确认工作区状态和待提交文件。

## 验证方法

- 只修改本仓库文件。
- 新增文档包含来源类型、问题场景、判断方法、操作步骤、验证方法、注意事项。
- 索引文件已同步更新。
- Markdown 空文件检查无输出。
- `git status` 中不包含业务项目或敏感文件。

## 注意事项

- 不确定是否安全的内容，先不要写入。
- 所有命令示例必须使用占位符或测试环境。
