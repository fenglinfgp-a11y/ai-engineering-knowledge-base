# AI Engineering Knowledge Base

这是一个用于长期沉淀 AI 辅助工程经验的个人技术知识库，重点记录合法开发、调试、部署、排错、项目交付、代码维护相关的通用方法。

## 知识库范围

- Codex 指令模板：分析、最小改动、部署排查、源码交付检查。
- Android/iOS 调试：公开文档、开源社区、合法测试环境下的构建与日志分析。
- 后端/前端/数据库：项目结构、接口排查、构建部署、备份恢复。
- 服务器部署：Ubuntu、Nginx、Docker、Java 服务、防火墙端口。
- 项目交付规范：backend、frontend、database 三件套，以及交付包清理规则。

## 内容规则

- 只整理公开资料、官方文档、开源项目经验、实战总结和我的项目经验。
- 不保存服务器密码、Token、密钥、客户资料、License 私钥、数据库真实数据。
- 不记录绕过登录、绕过支付、绕过授权、绕过风控、伪造订单、攻击第三方系统的内容。
- 每篇文档必须标注来源类型：官方文档 / 开源项目 / 实战总结 / 我的项目经验。
- 每篇排查文档尽量采用：问题场景 -> 判断方法 -> 操作步骤 -> 验证方法 -> 注意事项。
- 命令必须说明用途；危险命令必须注明风险。

## 推荐使用方式

1. 遇到问题先查 `99-index/技术索引.md`。
2. 常用命令查 `99-index/常用命令索引.md`。
3. 排错流程查 `99-index/问题排查索引.md`。
4. 新增经验时，按对应目录追加独立文档，不把无关主题混在一起。

## 这个知识库怎么用

- 查资料：先看 `99-index/技术索引.md`，再进入具体目录。
- 查命令：先看 `commands/`，涉及高风险操作时再看 `security/命令风险分级.md`。
- 查案例：先看 `cases/`，只复用脱敏后的判断方法、命令顺序和验证方式。
- 让 Codex 协作：优先复制 `00-codex-prompts/` 中的模板，并明确“只在本知识库内操作”。
- 记录新问题：先用 `.github/ISSUE_TEMPLATE/` 建立排查记录，再沉淀到 `cases/` 或对应技术目录。

## 新问题怎么沉淀

推荐流程：

1. 记录问题现象、环境背景和已执行操作。
2. 收集日志，先脱敏真实 IP、域名、账号、Token、客户资料和数据库真实数据。
3. 按 `workflows/问题排查到案例归档流程.md` 完成排查、修复和验证。
4. 把可复用经验写入 `cases/`，命令必须说明用途，危险命令必须标注风险。
5. 更新 `99-index/技术索引.md`、`99-index/问题排查索引.md`，命令类内容同步更新 `99-index/常用命令索引.md`。
6. 执行 `find . -name "*.md" -size 0` 和 `git status`，确认没有空文档和无关改动。

## 通过 GitHub Desktop 推送

1. 打开 GitHub Desktop，选择本仓库目录：`/Users/dora/Documents/ai-engineering-knowledge-base`。
2. 检查 Changes 面板，只应出现知识库相关 Markdown、索引或模板文件。
3. 填写提交说明，或使用 Codex 已完成的本地 commit。
4. 点击 `Push origin` 同步到 GitHub。
5. 推送后在 GitHub 仓库页面确认最新 commit 已出现。

命令行只做本地检查时可使用：

```bash
git status --short
```

用途：确认工作区是否干净，以及是否还有未提交文件。

```bash
git log --oneline -1
```

用途：查看本地最新 commit，便于和 GitHub 页面核对。

## 不建议保存的内容

- 服务器真实 IP、登录账号、密码、Token、SSH 私钥、License 私钥。
- 客户资料、合同、订单、支付记录、真实手机号、邮箱、地址。
- 真实数据库导出、业务表真实数据、生产日志原文。
- 第三方账号、授权凭据、风控细节、绕过登录/支付/授权/风控的方法。
- 闭源项目完整源码、客户项目业务代码、不可公开的商业配置。

## 问题场景

长期使用 AI 辅助开发时，经验容易散落在聊天记录、项目目录和临时笔记中，后续复用成本高。

## 判断方法

- 如果是通用调试、部署、交付经验，适合沉淀到本仓库。
- 如果包含客户资料、真实密钥、闭源业务代码，不适合沉淀到本仓库。
- 如果是一次性业务逻辑，不应写成通用知识文档。
