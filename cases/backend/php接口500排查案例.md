# PHP 接口 500 排查案例

来源类型：官方文档 / 实战总结

## 问题现象

访问 PHP 后端接口返回 500，前端只显示系统错误或请求失败。

## 环境背景

- PHP：`<php-version>`
- Web 服务：Nginx + PHP-FPM
- 项目目录：`/var/www/<app-name>`
- 接口：`/api/<example-endpoint>`

## 初步判断

优先查看 PHP fatal error、框架日志、依赖缺失、文件权限和环境变量。

## 排查命令

```bash
curl -i http://127.0.0.1/api/<example-endpoint>
```

用途：查看接口状态码、响应头和响应体。

```bash
tail -n 200 /var/www/<app-name>/storage/logs/app.log
```

用途：查看项目应用日志。风险：日志可能包含敏感字段，外发前必须脱敏。

```bash
php -v
```

用途：确认 PHP 版本。

```bash
php -m
```

用途：查看已启用 PHP 扩展。

## 关键日志

```text
PHP Fatal error: Uncaught Error: Class "<ClassName>" not found
```

含义：依赖或自动加载异常，常见于 Composer 依赖未安装或缓存未更新。

## 解决步骤

1. 用 `curl` 复现接口。
2. 查应用日志中的第一处 fatal error。
3. 确认依赖是否安装。
4. 确认 `.env` 测试配置存在且不包含真实密钥。
5. 修复后重复请求验证。

## 验证方式

- 接口不再返回 500。
- 应用日志不再出现同类 fatal error。
- 前端调用接口返回预期结构。

## 可复用经验

PHP 500 不要只看 Nginx 状态码，要看 PHP 应用日志和 PHP-FPM 日志。

## 禁止事项

- 不把真实 `.env`、数据库密码或客户请求参数写入文档。
- 不用关闭错误处理或跳过鉴权掩盖问题。
- 不复制闭源业务代码。
