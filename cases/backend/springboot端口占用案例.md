# Spring Boot 端口占用案例

来源类型：官方文档 / 实战总结

## 问题现象

Spring Boot 启动失败，日志提示端口 `<port>` 已被占用。

## 环境背景

- 服务端口：`<port>`
- 应用：`<app-name>`
- 部署方式：jar 或 systemd

## 初步判断

先确认端口被哪个进程占用，再判断是旧版本未停止、重复部署，还是端口配置冲突。

## 排查命令

```bash
ss -lntp | grep <port>
```

用途：查看占用目标端口的进程。

```bash
ps -fp <pid>
```

用途：查看进程详细信息，确认是否为同一应用。

```bash
sudo systemctl status <service-name>
```

用途：查看 systemd 服务状态。

```bash
sudo systemctl stop <old-service-name>
```

用途：停止旧服务。风险：会中断旧服务，执行前确认该服务确实应停止。

## 关键日志

```text
Port <port> was already in use
```

含义：应用绑定端口失败，进程无法启动 Web 服务。

## 解决步骤

1. 用 `ss` 找到占用端口的进程。
2. 判断是否为旧版本服务。
3. 如果是旧服务，按发布流程停止或替换。
4. 如果是端口配置冲突，调整其中一个服务端口。
5. 重新启动并验证健康检查。

## 验证方式

- 新服务启动日志显示成功。
- `ss -lntp` 中目标端口只对应预期进程。
- 接口健康检查返回 200。

## 可复用经验

端口占用不是简单 kill 进程，先确认进程归属和业务影响。

## 禁止事项

- 不直接 `kill -9` 未确认进程。
- 不记录真实服务名、内网 IP 或客户路径。
- 不通过随机改端口逃避网关配置同步。
