# MySQL 日志表过大清理案例

来源类型：官方文档 / 实战总结

## 问题现象

`<log_table>` 表数据量过大，磁盘告警，查询和备份变慢。

## 环境背景

- 表：`<log_table>`
- 时间字段：`created_at`
- 保留周期：`<retention-days>` 天

## 初步判断

先确认保留规则、时间字段索引、可删除范围和备份策略，再执行分批清理。

## 排查命令

```sql
SELECT COUNT(*) FROM <log_table>;
```

用途：查看日志表总量。

```sql
SELECT COUNT(*) FROM <log_table> WHERE created_at < '2026-01-01';
```

用途：删除前确认影响范围。

```sql
DELETE FROM <log_table> WHERE created_at < '2026-01-01' LIMIT 1000;
```

用途：分批删除旧日志。风险：会删除数据，必须先备份并确认条件。

## 关键日志

```text
Lock wait timeout exceeded
```

含义：清理操作可能与业务写入冲突，需要降低批量或调整低峰执行。

## 解决步骤

1. 备份表结构和必要数据。
2. 统计清理范围。
3. 分批删除，观察锁等待和业务日志。
4. 增加定期清理或归档策略。
5. 评估是否需要分区表。

## 验证方式

- 过期数据数量下降。
- 应用日志无 SQL 异常。
- 磁盘告警缓解。

## 可复用经验

日志表清理的关键不是一次删完，而是可恢复、可验证、可持续。

## 禁止事项

- 不执行无 WHERE 条件的 DELETE。
- 不记录真实日志内容。
- 不在业务高峰期大批量删除。
