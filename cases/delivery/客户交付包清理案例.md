# 客户交付包清理案例

来源类型：实战总结 / 我的项目经验

## 问题现象

交付包中混入日志、缓存、依赖目录、构建产物、真实配置或压缩备份。

## 环境背景

- 交付包：`<delivery-package>`
- 类型：backend / frontend / database
- 目标：交付可开发源码和必要说明

## 初步判断

先扫描敏感文件、缓存目录、构建产物和大型文件，再人工确认是否需要保留。

## 排查命令

```bash
find . -type f \( -name ".env" -o -name "*.pem" -o -name "*.key" -o -name "*.log" -o -name "*.sql" \)
```

用途：查找环境变量、私钥、日志和 SQL 文件。

```bash
find . -type d \( -name node_modules -o -name vendor -o -name target -o -name dist -o -name build \) -prune -print
```

用途：查找依赖目录和构建产物目录。

```bash
du -sh *
```

用途：查看顶层目录大小，发现异常大的目录。

## 关键日志

```text
发现 .env 或 *.pem 文件，需要人工确认并移除真实敏感信息
```

含义：交付包可能包含密钥或真实配置，不能直接交付。

## 解决步骤

1. 先备份原始交付包。
2. 扫描敏感文件和构建产物。
3. 用 `.env.example` 替代真实 `.env`。
4. 删除缓存、日志、依赖目录和真实备份。
5. 重新生成文件清单。

## 验证方式

- 交付包不含真实密钥和客户数据。
- README 能说明依赖安装和构建方式。
- 接收方可根据源码复现构建产物。

## 可复用经验

交付清理不是简单删大文件，而是保留可复现源码，移除敏感和可再生成内容。

## 禁止事项

- 不保存真实客户数据。
- 不提交真实 `.env`。
- 不使用高风险删除命令处理未确认路径。
