# MySQL 日志表清理案例

来源类型：实战总结

## 问题场景

MySQL 日志表持续增长，需要在保留必要数据的前提下安全清理或归档。

## 问题现象

日志表持续增长，查询变慢，磁盘空间接近告警阈值。

## 环境背景

- 数据库：MySQL `<version>`
- 表：`<log_table>`
- 时间字段：`created_at`
- 所有库名、表名、字段值均为脱敏占位符。

## 初步判断

优先判断是否有保留周期、是否可以归档、是否存在索引、是否能分批清理。

## 排查命令

```sql
SELECT COUNT(*) FROM <log_table>;
```

用途：查看日志表总行数。

```sql
SELECT COUNT(*) FROM <log_table> WHERE created_at < '2026-01-01';
```

用途：清理前统计将影响的数据量。

```sql
SHOW INDEX FROM <log_table>;
```

用途：确认时间字段是否有索引。

```sql
DELETE FROM <log_table> WHERE created_at < '2026-01-01' LIMIT 1000;
```

用途：分批删除过期日志。风险：会删除数据，执行前必须备份并确认条件。

## 关键日志

```text
The table '<log_table>' is full
```

含义：表或磁盘空间不足，需要先确认磁盘、表空间和日志增长策略。

## 解决步骤

1. 备份表结构和必要数据。
2. 用 `SELECT COUNT(*)` 验证清理范围。
3. 低峰期分批删除。
4. 观察业务日志和数据库负载。
5. 必要时增加保留策略或归档任务。

## 验证方式

- 清理后目标数据范围不再存在。
- 应用功能正常。
- 慢查询和磁盘告警缓解。
- 备份文件可用于测试库恢复。

## 可复用经验

日志表清理必须先备份、先 SELECT 验证范围，再分批 DELETE。大表长期方案应考虑归档、分区或冷热分离。

## 禁止事项

- 不保存真实数据库数据。
- 不执行无 WHERE 条件的 DELETE。
- 不在业务高峰期直接清理大表。

## 适用范围

适用于授权数据库维护和脱敏测试环境演练。

## 判断方法

如果没有备份、没有保留规则、没有清理范围确认，则不能进入删除步骤。
