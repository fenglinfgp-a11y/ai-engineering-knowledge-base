# Ubuntu 端口不通排查案例

来源类型：官方文档 / 实战总结

## 问题现象

服务在服务器本机能访问，但外部访问 `http://<server-ip>:<port>` 超时或连接失败。

## 环境背景

- 系统：Ubuntu `<version>`
- 服务端口：`<port>`
- 服务监听：`<bind-address>:<port>`
- 防火墙：UFW 或云厂商安全组

## 初步判断

优先判断服务是否监听、监听地址是否正确、本机防火墙和云安全组是否开放。

## 排查命令

```bash
ss -lntp | grep <port>
```

用途：查看服务是否监听目标端口，以及监听在 `127.0.0.1` 还是 `0.0.0.0`。

```bash
sudo ufw status numbered
```

用途：查看 UFW 防火墙规则和编号。

```bash
curl -i http://127.0.0.1:<port>
```

用途：从服务器本机验证服务是否可访问。

```bash
sudo ufw allow <port>/tcp
```

用途：开放 TCP 端口。风险：会增加公网暴露面，执行前确认端口确实需要对外开放。

## 关键日志

```text
Connection timed out
```

含义：常见于防火墙、安全组或网络链路阻断，不等同于应用服务报错。

## 解决步骤

1. 用 `ss` 确认服务是否监听。
2. 如果只监听 `127.0.0.1`，调整服务监听地址为需要的网卡地址。
3. 检查 UFW 规则。
4. 检查云安全组入站规则。
5. 外部重新访问端口验证。

## 验证方式

- 本机 `curl` 成功。
- 外部机器访问 `<server-ip>:<port>` 成功。
- 防火墙规则只开放必要端口。

## 可复用经验

端口不通要区分“服务没监听”“只监听本机”“防火墙拦截”“云安全组未开放”。

## 禁止事项

- 不随意开放数据库、Redis、管理面板到公网。
- 不记录真实服务器 IP。
- 不关闭全部防火墙规则来省事。
