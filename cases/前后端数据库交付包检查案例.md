# 前后端数据库交付包检查案例

来源类型：实战总结 / 我的项目经验

## 问题场景

接收或交付项目源码时，需要判断交付包是否完整、可开发、可部署且不含敏感信息。

## 问题现象

收到交付包后无法判断是否包含可继续开发和部署所需的 backend、frontend、database 三件套。

## 环境背景

- 交付包：`<project-delivery-package>`
- 后端技术：`<backend-stack>`
- 前端技术：`<frontend-stack>`
- 数据库：`<database-type>`

## 初步判断

优先检查源码入口、依赖声明、启动说明、环境变量示例、数据库结构和敏感文件。

## 排查命令

```bash
find . -maxdepth 3 -type f | sort
```

用途：查看交付包前三层文件结构。

```bash
find . -maxdepth 3 -name "package.json" -o -name "pom.xml" -o -name "composer.json"
```

用途：查找常见依赖声明文件。

```bash
find . -type f \( -name ".env" -o -name "*.pem" -o -name "*.key" -o -name "*.sql" \)
```

用途：查找需要重点检查的环境变量、密钥和 SQL 文件。

```bash
du -sh *
```

用途：查看顶层目录大小，发现异常大的日志、缓存或依赖目录。

## 关键日志

```text
缺少 database/schema.sql 或迁移文件
```

含义：数据库结构不可复现，交付包不满足完整开发部署要求。

## 解决步骤

1. 补充 backend 启动说明和 `.env.example`。
2. 补充 frontend 安装、开发、构建命令。
3. 补充 database 结构 SQL 或迁移文件。
4. 清理真实密钥、缓存、日志和真实数据。
5. 更新 README 的部署顺序。

## 验证方式

- 后端能在测试环境启动。
- 前端能构建。
- 空数据库能导入结构。
- 敏感文件扫描没有真实密钥。

## 可复用经验

源码交付不是把整个目录压缩给对方，而是交付可复现的源码、配置示例、数据库结构和运行说明。

## 禁止事项

- 不交付真实 `.env`。
- 不交付客户真实数据。
- 不把构建产物当成源码。

## 适用范围

适用于合法项目交付、接手维护和源码完整性检查。

## 判断方法

如果缺少依赖声明、启动入口、数据库结构或配置示例，则不能判断为完整可开发源码。
