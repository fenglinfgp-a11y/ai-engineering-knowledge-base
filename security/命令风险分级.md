# 命令风险分级

来源类型：实战总结 / 我的项目经验

## 问题场景

知识库中包含大量排查命令，需要区分只读、安全变更和高风险操作。

## 适用范围

适用于所有命令文档、案例和部署流程。

## 判断方法

- 只读命令：只查看状态，不改变系统。
- 低风险命令：只影响本地临时产物或测试环境。
- 中风险命令：重启、reload、修改配置，会影响服务。
- 高风险命令：删除、覆盖、清库、强制推送、开放公网端口。

## 操作步骤

| 等级 | 类型 | 示例 | 写法要求 |
| --- | --- | --- | --- |
| 低 | 查看状态 | `git status`、`ss -lntp`、`nginx -t` | 说明用途即可 |
| 中 | 重载/重启 | `systemctl restart <service>`、`nginx -s reload` | 标注可能中断服务 |
| 高 | 删除/覆盖/写库 | `DELETE ...`、`mysql < backup.sql`、`ufw allow <port>` | 必须写风险、前置确认、验证方式 |
| 禁止 | 极端破坏或绕过 | 全盘删除、force push、绕过授权 | 不写入知识库 |

## 相关命令

```bash
git diff --stat
```

用途：查看修改范围，避免误把高风险命令写入无关文档。

```bash
grep -R "风险：" . --include="*.md"
```

用途：检查危险命令是否标注风险。

## 验证方法

- 高风险命令旁边明确写了“风险”。
- 命令前有确认条件。
- 命令后有验证方法。

## 注意事项

- 不提供极端破坏性命令示例。
- 不鼓励在生产环境直接执行写操作。
- 所有生产变更先备份、再验证、再执行。
