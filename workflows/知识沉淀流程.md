# 知识沉淀流程

来源类型：实战总结 / 我的项目经验

## 问题场景

遇到新的工程问题后，需要把一次性排查经验沉淀成以后能复用的知识文档。

## 适用范围

- 服务器部署、后端接口、前端构建、数据库维护、Android/iOS 调试、源码交付。
- 只适用于合法开发、部署、排查、交付和授权测试场景。

## 判断方法

- 能被复用的问题适合沉淀。
- 只属于客户私有业务逻辑的问题不直接沉淀。
- 涉及真实日志、账号、IP、Token、数据库数据时，必须先脱敏。

## 操作步骤

1. 记录问题现象：只写通用现象，不写客户名称。
2. 收集环境信息：使用版本号、系统类型、端口占位符。
3. 保存关键日志：只保留必要片段，并替换敏感信息。
4. 梳理排查路径：按“命令 -> 用途 -> 结果 -> 判断”记录。
5. 写解决步骤：只写合法修复，不写绕过类做法。
6. 写验证方式：说明如何确认问题已修复。
7. 写入 `cases/` 或对应主题目录。
8. 更新 `99-index/` 三个索引。
9. 执行 Markdown 空文件检查。
10. 提交 commit，必要时用 GitHub Desktop 推送。

## 相关命令

```bash
find . -name "*.md" -size 0
```

用途：检查是否存在空 Markdown 文档。

```bash
git status --short
```

用途：查看当前修改文件。

```bash
git diff --stat
```

用途：查看变更范围，确认没有误改无关文件。

## 验证方法

- 新文档能从 `99-index/技术索引.md` 找到。
- 文档包含来源类型、问题场景、操作步骤、验证方法和注意事项。
- 文档不包含真实敏感信息。

## 注意事项

- 沉淀的不是客户问题本身，而是通用排查方法。
- 所有示例值使用 `<placeholder>` 或 `example.com`。
- 不确定是否能公开的内容先不写。
