# 问题排查到案例归档流程

来源类型：实战总结 / 我的项目经验

## 问题场景

一次问题排查完成后，需要把现场过程转成脱敏案例，放入 `cases/` 并保持索引可查。

## 适用范围

适用于 Nginx 502、Docker 启动失败、Spring Boot 启动失败、MySQL 慢 SQL、Android 崩溃、源码交付缺失等案例归档。

## 判断方法

- 如果问题有明确现象、命令、日志、解决步骤和验证方式，就适合归档。
- 如果问题依赖客户私有业务逻辑，归档时只保留通用排查框架。

## 操作步骤

1. 问题：写清楚现象，例如“接口 500”“端口不通”“APK 安装失败”。
2. 日志：只保留能说明问题的几行，并替换敏感字段。
3. 排查：列出命令，每条命令写用途。
4. 修复：写最小修复步骤，危险操作标注风险。
5. 验证：写复测方法，例如状态码、服务状态、日志关键字消失。
6. 脱敏：检查 IP、域名、账号、Token、客户名、数据库数据。
7. 写入 cases：按类别放到 `cases/server/`、`cases/backend/` 等目录。
8. 更新索引：把案例加入技术索引、命令索引、问题排查索引。
9. 提交：运行检查后 commit。
10. 同步：用 GitHub Desktop 或已配置凭据的 Git 推送。

## 相关命令

```bash
grep -R "TODO\\|TBD" cases --include="*.md"
```

用途：查找案例中未完成的占位内容。

```bash
grep -R "password\\|token\\|secret\\|PRIVATE KEY" . --include="*.md"
```

用途：粗略扫描敏感词。风险：可能输出敏感内容，结果不要外发。

## 验证方法

- `cases/` 中新增了对应脱敏案例。
- 99-index 三个索引都有入口。
- Markdown 空文件检查无输出。

## 注意事项

- 不要把完整生产日志粘进案例。
- 不要把真实 SQL 结果写进案例。
- 不写绕过登录、支付、授权、风控的内容。
